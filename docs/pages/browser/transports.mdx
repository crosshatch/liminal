# Browser and Worker Transports

Liminal is not limited to WebSockets.

Two transport families are available:

- `Client.layerSocket(...)` for server-backed actors over WebSocket
- `Client.layerWorker(...)` for browser-local actors over workers or `MessagePort`

`liminal-browser` adds `ActorRegistry.make(...)`, which lets you host actors locally inside the browser and talk to them
through the same client protocol.

## `Client.layerSocket(...)`

Use sockets when the actor lives on the server.

```ts
import { BrowserSocket } from "@effect/platform-browser"
import { Layer } from "effect"
import { Client } from "liminal"

const ClientsLive = Layer.mergeAll(
  Client.layerSocket({
    client: LobbyClient,
    url: "/session",
  }),
  Client.layerSocket({
    client: ChatClient,
    url: "/session",
  }),
).pipe(Layer.provide(BrowserSocket.layerWebSocketConstructor))
```

`layerSocket(...)` requires a `Socket.WebSocketConstructor` in the environment. In the browser that comes from
`BrowserSocket.layerWebSocketConstructor`; on the server (for example, in tests) you can use
`NodeSocket.layerWebSocketConstructor` from `@effect/platform-node`.

## `Client.layerWorker(...)`

Use worker transport when the actor lives in a worker, a `MessagePort`, or another browser-local runtime.

```ts
import { Layer } from "effect"
import { Client } from "liminal"

const NotificationLive = NotificationAccumulatorLive.pipe(
  Layer.provideMerge(Client.layerWorker({ client: NotificationClient }).pipe(Layer.provide(NotificationWorker.layer))),
)
```

`layerWorker(...)` needs a `Worker.WorkerPlatform` and `Worker.Spawner` in the environment (both from
`effect/unstable/workers`). The exact provider depends on whether you are spawning a real worker, bridging through a
platform adapter, or forwarding a `MessagePort`.

## Host actors in the browser with `ActorRegistry.make(...)`

`ActorRegistry.make(...)` from `liminal-browser` runs actors inside the current JavaScript realm. It wires up the same
client protocol, event stream, and method handlers, but without any network layer — each client reaches the actor
through a `MessagePort` instead of a socket. Multiple clients can coexist, and multiple actor-names are supported per
registry (keyed the same way Cloudflare Durable Objects are).

This is useful for:

- single-player or offline-first experiences that share code with the networked path
- web-worker-hosted actors that own local state (notifications, caches, background sync)
- tests that want to exercise handlers without a server

The registry is driven by a `Stream<Introduction>`. Each `Introduction` carries the `MessagePort` for a new client, the
actor-name to route it to, and the initial `attachments`:

```ts
import { Stream } from "effect"
import type { Introduction } from "liminal-browser/ActorRegistry"

import type { NotificationActor } from "./NotificationActor.ts"

type NotificationIntroduction = Introduction<
  typeof NotificationActor.definition.name,
  typeof NotificationActor.definition.attachments
>

declare const introductions: Stream.Stream<NotificationIntroduction>
```

Where the stream comes from depends on how clients reach the registry. For a same-realm setup, use a `MessageChannel`
per client:

```ts
import { Effect, Stream } from "effect"
import { ActorRegistry } from "liminal-browser"

import { handlers } from "./handlers.ts"
import { NotificationActor } from "./NotificationActor.ts"
import { onConnect } from "./onConnect.ts"

const channel = new MessageChannel()
const introductions = Stream.make({
  port: channel.port1,
  name: "singleton",
  attachments: {},
})

const program = ActorRegistry.make({
  actor: NotificationActor,
  handlers,
  onConnect,
  introductions,
})
```

For a `SharedWorker`, adapt the `connect` event:

```ts
import { Stream } from "effect"

declare const self: SharedWorkerGlobalScope

const introductions = Stream.fromEventListener<MessageEvent>(self, "connect").pipe(
  Stream.map((e) => ({
    port: e.ports[0]!,
    // Name/attachments come from whatever handshake your caller uses — typically
    // the first message on the port, or info in the URL the SharedWorker was opened with.
    name: "singleton" as const,
    attachments: {},
  })),
)
```

Pair the registry with `Client.layerWorker(...)` on the consuming side to reach the hosted actors through the normal
client surface.
