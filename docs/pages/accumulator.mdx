# Accumulator

An `Accumulator` is a typed state container driven by events. It reduces incoming events into long-lived client state,
giving UI code a stable state stream instead of forcing every component to understand the full event log.

If you are building anything larger than a toy screen, this is the most important guide after the quickstart.

## Define an accumulator

An accumulator is a tagged service parameterized by its state type.

```ts
import { Accumulator } from "liminal"

import type { LobbyEvent } from "./LobbyClient.ts"

interface LobbyState {
  readonly currentRoomId: string | undefined
}

export class LobbyAccumulator extends Accumulator.Service<LobbyAccumulator, LobbyState>()("LobbyAccumulator") {}
```

The state is a plain TypeScript type. There is no schema config — you control the shape directly.

## Extract initial state from the first event

The `initial` function inspects each event until it finds one that produces the starting state.

```ts
import { Effect, Option } from "effect"

const initial = (event: LobbyEvent) => {
  if (event._tag === "LobbySnapshot") {
    return Effect.succeed(
      Option.some<LobbyState>({
        currentRoomId: event.currentRoomId,
      }),
    )
  }
  return Effect.succeed(Option.none())
}
```

Until `initial` returns `Some`, the accumulator is not yet active and no reductions occur.

## Write the reduce function

The `reduce` function folds each subsequent event into the current state.

```ts
const reduce = (event: LobbyEvent) => (state: LobbyState) => {
  switch (event._tag) {
    case "RoomJoined":
      return Effect.succeed({ ...state, currentRoomId: event.roomId })
    case "SessionEnded":
      return Effect.succeed({ ...state, currentRoomId: undefined })
    default:
      return Effect.succeed(state)
  }
}
```

## Use `reducer` for type-safe per-event reducers

The `reducer` helper narrows the event type to a single tag, which is useful when composing reducers.

```ts
const RoomJoined = LobbyAccumulator.reducer<LobbyEvent>()(
  "RoomJoined",
  ({ roomId }) =>
    (state) =>
      Effect.succeed({ ...state, currentRoomId: roomId }),
)

const SessionEnded = LobbyAccumulator.reducer<LobbyEvent>()(
  "SessionEnded",
  () => (state) => Effect.succeed({ ...state, currentRoomId: undefined }),
)
```

Each call returns the reducer function unchanged — the value is in the type narrowing, not runtime behavior.

## Build the accumulator layer

Combine the source stream, initial function, and reduce function into a layer.

```ts
export const LobbyAccumulatorLive = LobbyAccumulator.layer({
  source: LobbyClient.events,
  initial,
  reduce,
})
```

That layer starts the reduction loop and exposes:

- `LobbyAccumulator.get` — an `Effect` that reads the current state
- `LobbyAccumulator.stream` — a `Stream` of state changes

## Consume state

UI code reads the accumulator's current value or subscribes to its stream.

```ts
Effect.gen(function* () {
  // ...

  // Read the current state
  const state = yield* LobbyAccumulator.get

  // Subscribe to state changes
  const stateStream = LobbyAccumulator.stream
})
```

Atoms or signals in the UI layer can subscribe to the stream directly:

```ts
export const lobbyStateAtom = atomRuntime.atom(LobbyAccumulator.stream)
```

## Snapshot-on-connect plus delta events

The accumulator design depends on a deliberate event pattern:

1. `onConnect` sends a snapshot event such as `LobbySnapshot` or `SessionStarted`
2. later handlers emit delta events such as `RoomJoined` or `SessionEnded`
3. `initial` extracts state from the snapshot, then `reduce` folds deltas

That is why `onConnect` is so important. Without a snapshot event, a reconnecting UI would need a separate imperative
fetch path.

## Real multi-client pattern

Larger apps combine `Audition` and `Accumulator` together.

An app-wide accumulator listens to both lobby and chat events:

```ts
interface AppState {
  readonly roomId: string | undefined
  readonly userId: string | undefined
  readonly members: Array<{
    readonly userId: string
    readonly displayName: string
  }>
}

export class AppAccumulator extends Accumulator.Service<AppAccumulator, AppState>()("AppAccumulator") {}
```

And its live layer uses the merged event stream, with `initial` extracting the first snapshot:

```ts
const source = audition.events.pipe(Stream.forever)

const AppAccumulatorLive = AppAccumulator.layer({
  source,
  initial: (event) => {
    if (event._tag === "SessionStarted") {
      return Effect.succeed(
        Option.some<AppState>({
          roomId: event.roomId,
          userId: event.userId,
          members: event.members,
        }),
      )
    }
    return Effect.succeed(Option.none())
  },
  reduce,
})
```

The important part is that the first snapshot event becomes the accumulator's initial value.

## When to use `Accumulator`

Use `Accumulator` when:

- multiple screens care about the same evolving state
- reconnects should replay state from events
- `onConnect` already emits a snapshot event
- you want a stable state stream derived from events

Skip it when a screen only needs one or two direct method calls and no long-lived local state.
